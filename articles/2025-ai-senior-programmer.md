# 2025年は、AI上級プログラマーが誕生した年

## はじめに - コードを書く時代から、タスクを任せる時代へ

2025年は、プログラミングの歴史において大きな転換点として記憶されるでしょう。

それは、**AIが「ペアプログラマー」から「上級プログラマー」へと進化した年**です。

これまでのAIは、コードの補完や簡単な質問への回答が主な役割でした。しかし、Cursor、Claude Code、Antigravityといった新世代のツールは、まったく異なるアプローチを取ります。

**彼らは、タスクを理解し、プロジェクト全体を見渡し、一貫性のある変更を複数のファイルにまたがって実行できる、本物のエージェントです。**

まるで、経験豊富な上級プログラマーが隣にいて、あなたの指示を理解し、最適な実装を提案してくれるかのように。

## Cursor + Claude: 最強の組み合わせ

このCookbookでは、**Cursor + Claude**の組み合わせを強く推奨しています。

なぜこの組み合わせなのか？

### Cursorの3つの強力な機能

**1. Composer（Ctrl+I）- プロジェクト全体への指示**

```
lib/配下にhome_page.dartを作成して、以下の機能を実装してください：
1. TODOリストの表示
2. 新しいTODOを追加するボタン
3. 完了したTODOをチェックする機能
```

この指示だけで、Composerは：
- 適切なファイル構造を作成
- 必要なインポート文を追加
- 既存のコードとの整合性を保つ
- Material Designに沿ったUIを生成

**2. Chat（Ctrl+L）- 相談と学習**

```
このエラーの原因を教えてください：
The method 'setState' isn't defined for the class 'MyWidget'
```

上級プログラマーのように：
- エラーの原因を的確に説明
- 解決方法を複数提示
- なぜそのエラーが起きたのか、根本原因を教える

**3. Inline Edit（Ctrl+K）- 素早い修正**

コードを選択して「このボタンを大きくして、角を丸くしてください」と指示するだけで、適切な変更が適用されます。

### なぜClaudeなのか？

2025年12月時点で、ClaudeはFlutter開発に最も適したAIモデルです。

**Claude Opus 4.5 / Sonnet 4.5の強み：**

1. **ハルシネーションが少ない**
   - 「分からないことは分からない」と言える誠実さ
   - 存在しないAPIやメソッドを作り出さない
   - 最新のドキュメントを読み解く能力

2. **構造化コードに強い**
   - Flutterのツリー構造を正確に理解
   - ファイル間の依存関係を把握
   - 大規模なリファクタリングでもコードを壊さない

3. **保守的で慎重**
   - 必要な箇所だけを最小限に修正
   - 予期せぬ破壊的変更が少ない
   - ベストプラクティスに沿ったコード

4. **説明が丁寧**
   - なぜこの構造にするのか
   - なぜこの依存関係を選ぶのか
   - Flutter公式ガイドラインとの対応

**他のモデルとの比較（2025年12月時点）：**

| モデル | コーディング安定性 | Flutter/UI | 特徴 |
|--------|------------------|-----------|------|
| **Claude Opus 4.5** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 最高峰の理解力、複雑な実装に最適 |
| **Claude Sonnet 4.5** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 高速で安定、簡易タスクに最適 |
| ChatGPT GPT-5.1 | ⭐⭐⭐⭐ | ⭐⭐⭐ | 速度は速いが最新Flutter機能に弱い |
| Gemini 2.0 Flash Thinking | ⭐⭐ | ⭐⭐⭐⭐ | 高速で強力だが、実装の安定性に課題 |

### Cursor + Claudeの実際

上級プログラマーとペアプログラミングしているような感覚：

**あなた：** 「@lib/main.dart を見て、このアプリの構造を説明してください」

**Claude：** 「このアプリはStatefulWidgetを使った標準的な構造です。`_MyHomePageState`で状態管理を行い、`setState`で画面を更新しています。現在はカウンター機能のみですが、この構造を拡張してTODO機能を追加することができます...」

**あなた：** 「@pubspec.yaml に http パッケージを追加してください」

**Claude：** 「了解しました。依存関係セクションに `http: ^1.2.0` を追加します。これは最新の安定版です...」

まさに、**経験豊富な上級プログラマーが隣にいる**ような開発体験です。

## エージェント時代に必要な能力

実際にAIエージェントと開発して分かったことがあります。

**コーディングエージェントに必要なのは、難しい問題を解く能力ではない。**

一番困るのは：
- ハルシネーションを起こしてコードを書き換える
- 古い記憶を「正解」として押し付ける
- 最新のAPIを無視する
- 既存コードを破壊する

エージェントに必要なのは：
- **記憶に頼らず新しいドキュメントを読み解く能力**
- **「分からないことは分からない」と言える誠実さ**
- **保守的で慎重なコード変更**
- **一貫性のある大規模修正**

だからこそ、Claudeが選ばれているのです。

## AI x Flutter Cookbookの誕生

このCookbookは、**「Cursor + Claudeを使えば、どこまでアプリ開発ができるのか？」**という問いから始まりました。

数百回にわたるClaudeとの対話、実際のアプリ開発、そして失敗と成功の繰り返し。その全てがこのCookbookに詰まっています。

### 特徴1: エージェント時代の開発手法

**従来のAI（2024年まで）：**
- コードの一部を補完
- 質問に答える
- 一緒にコードを書く「ペアプログラマー」

**2025年のAI（Cursor + Claude）：**
- プロジェクト全体を理解
- 複数ファイルにまたがる変更を一貫性を持って実行
- タスクを任せられる「上級プログラマー」

**開発スタイルの変化：**

```
【旧】あなた: 「このコードをどう書けばいいですか？」
     AI: 「こう書いてください」
     あなた: [コードを書く]

【新】あなた: 「TODOアプリのホーム画面を作ってください」
     Claude: [ファイルを作成し、コードを実装し、既存ファイルと統合]
     あなた: [結果を確認し、必要に応じて調整を依頼]
```

**あなたは設計と判断に集中できます。**

### 特徴2: シンプルさの追求

実際の開発で気づいたことがあります。

**高機能なパッケージほど、AIはハルシネーションを起こす。**

Riverpod、BLoC、GetXのような高機能な状態管理ライブラリは：
- 複雑なAPI
- 頻繁なバージョン変更
- 抽象化されたパターン

これらが、AIの「古い記憶」と衝突し、動かないコードや既存コードの破壊を引き起こします。

**このCookbookが推奨するのは：**

- **小さな自己完結型Widget** - 1ファイルで完結
- **setState中心の状態管理** - シンプルで理解しやすい
- **依存関係の最小化** - AIが壊しにくい構造

**「シンプルであることが、人間にもAIにも優しい」**

これは、実際のアプリ開発で得た重要な教訓です。

### 特徴3: 実践的なトラブルシューティング

AI開発には特有のトラブルがあります。

このCookbookには、「AIとの開発で起こる典型的なトラブルと対処法」が含まれています：

1. AIが古いコードを生成する → `@Docs`で公式ドキュメントを参照させる
2. AIが間違った依存関係を追加する → バージョンを明示的に指定
3. AIがハルシネーションを起こす → 小さく実装して確認
4. AIが既存コードを壊す → Gitで必ずバージョン管理
5. 望む結果が得られない → 具体的で明確なプロンプト

**上級プログラマーとうまく働くためのノウハウ**が詰まっています。

## Cursor + Claudeの実践例

### 例1: 新しい画面を作成

**Composer（Ctrl+I）で：**
```
lib/screens/settings_page.dart を作成して、以下の設定項目を持つ画面を作ってください：
1. ダークモード切り替え
2. 通知のオン/オフ
3. 言語選択（日本語/英語）

Material Designに沿って、SharedPreferencesで設定を保存してください。
```

**結果：**
- 適切なファイル構造
- 完全に動作するコード
- 設定の永続化
- 既存コードとの統合

まるで、**上級プログラマーに頼んだかのような完成度**。

### 例2: エラー解決

**Chat（Ctrl+L）で：**
```
以下のエラーが出ています。原因と解決方法を教えてください：

[エラーメッセージをペースト]
```

**Claudeの回答：**
- エラーの根本原因を説明
- 複数の解決方法を提示
- なぜそのエラーが起きたのか、背景知識も教える
- ベストプラクティスも提案

**まるで、経験豊富な先輩プログラマーに相談したかのよう。**

### 例3: リファクタリング

**@マークでファイルを参照：**
```
@lib/home_page.dart
@lib/widgets/todo_item.dart

この2つのファイルを見て、重複コードを共通化してください。
既存の機能を壊さないように注意してください。
```

**Claudeは：**
1. 両方のファイルを理解
2. 重複部分を特定
3. 共通ウィジェットを作成
4. 両方のファイルを更新
5. インポート文も適切に追加

**一貫性のある、プロフェッショナルなリファクタリング。**

## なぜFlutterなのか？

Flutterを選んだ理由：

1. **一つのコードで全プラットフォーム**
   - iOS、Android、Web、Windows、macOS、Linux
   - コード共有率90%以上

2. **日本のiPhoneシェアの高さ**
   - iOS開発が重要
   - FlutterならiOSも対応可能

3. **AIとの相性の良さ**
   - 構造化されたコード（ウィジェットツリー）
   - 豊富な公式ドキュメント
   - Claudeが正確に理解できる

4. **シンプルな状態管理**
   - StatefulWidget + setState
   - 複雑なライブラリ不要
   - AIが壊しにくい
